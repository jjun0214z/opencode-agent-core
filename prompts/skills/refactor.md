# Skill: refactor

## Trigger
`/refactor [path]`

## What This Is
기능 변경 없이 코드 구조를 개선한다.
동작 보존이 전제 조건이다. 동작이 바뀌면 리팩터링이 아니다.

---

## Phase 0: Scope & Constraint

1. 대상 경로/파일 확인
2. 현재 테스트 커버리지 확인 — 없으면 리팩터링 전 `/test` 먼저 권고
3. 리팩터링 목표 명확화:
   - 중복 제거 (DRY)
   - 복잡도 감소
   - 네이밍 개선
   - 모듈 분리

---

## Phase 1: Read & Analyze

변경 전 반드시:
1. 대상 파일 전체 읽기
2. 현재 패턴/컨벤션 파악
3. 호출자(caller) 파악 — 시그니처 변경 시 영향 범위
4. 테스트 존재 여부 확인

---

## Phase 2: Refactor

원칙:
- **한 번에 하나의 목표만** — 중복 제거와 모듈 분리를 동시에 하지 않는다
- **외부 동작 불변** — public API, 반환값, 부수효과 유지
- **기존 패턴 따름** — 새 패턴 도입 금지
- 3번 미만 반복 코드는 추상화 금지

---

## Phase 3: Verify

- [ ] 동작 변경 없음 (테스트 통과)
- [ ] 타입 에러 없음
- [ ] 호출자 파일 영향 없음
- [ ] 전후 비교 요약 제시

---

## Anti-patterns
- 기능 추가/변경과 리팩터링 동시 수행 금지
- 테스트 없는 코드 대규모 리팩터링 금지
- 요청 범위 밖 파일 수정 금지
- `git commit` / `git push` 금지
